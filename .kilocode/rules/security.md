# Правила безопасности для проектов

## ЗАПРЕЩЕНО: Файлы .env в Git

**ПРАВИЛО №1**: Никогда не добавлять файлы .env в систему контроля версий Git

### Что запрещено:
- `.env`
- `.env.local`
- `.env.production`
- `.env.development`
- `.env.staging`
- Любые файлы с токенами, паролями, ключами API

### Почему это критически важно:
1. **Утечка данных**: Токены и ключи станут доступны всем с доступом к репозиторию
2. **Компрометация**: Злоумышленники могут получить доступ к вашим сервисам
3. **Финансовые потери**: Особенно для платных API и сервисов
4. **Репутационный ущерб**: Утечка данных клиентов и партнеров

### Правильная практика:
1. **Всегда добавлять .env в .gitignore** ПЕРЕД первым коммитом
2. **Создавать .env.example** с примерами переменных (без реальных значений)
3. **Использовать переменные окружения** платформы развертывания
4. **Проверять историю коммитов** перед пушем

### Если .env попал в Git:
1. НЕМЕДЛЕННО удалить из истории:
   ```bash
   git filter-repo --invert-paths --path .env --force
   git push origin --force --all
   ```
2. Отозвать скомпрометированные токены
3. Сгенерировать новые ключи
4. Обновить переменные на серверах

### Автоматическая защита:
- VS Code настроен скрывать файлы .env
- .gitignore включает все варианты .env файлов
- Проверять git status перед каждым коммитом
